# 财务报销 Agent — 系统设计文档

> 版本：v0.1 | 作者：yangyinmiao | 更新日期：2025-05

---

## 一、背景与痛点

根据行业调研，企业财务报销场景存在以下核心痛点：

| 痛点 | 具体表现 | 影响 |
|------|----------|------|
| **流程繁琐** | 多级审批、纸质单据、线下传递 | 员工体验差，财务效率低 |
| **政策不清晰** | 员工不清楚哪些能报、上限多少 | 大量无效申请，退回率高 |
| **发票管理混乱** | 手工录入、易丢失、难追溯 | 对账困难，合规风险高 |
| **缺乏监督机制** | 无实时风控，靠人工抽查 | 资金滥用风险，审计困难 |
| **数据孤岛** | 报销数据与财务系统不联通 | 无法做统计分析和预算管控 |

**本系统目标：** 用 AI Agent 打通"提交 → 识别 → 校验 → 审批 → 归档"全链路，解决以上痛点。

---

## 二、系统目标

- 支持 **500人规模** 企业日常报销需求
- 发票识别准确率 > 95%
- 单次识别响应时间 < 5秒
- 审批流程从平均 3天 压缩至 **当天完成**
- 支持企业微信/钉钉消息通知

## 三、功能模块

### 3.1 当前版本（MVP）

```
✅ 发票上传（JPG / PNG / PDF）
✅ GPT-4o Vision 自动识别发票信息
✅ 报销规则校验（金额上限、必填字段）
✅ 生成报销摘要，支持下载
```

### 3.2 完整版功能规划

#### 员工端
- 上传发票（单张/批量）
- 填写报销说明、关联项目
- 查看审批进度
- 历史报销记录查询

#### 财务/审批端
- 待审批列表（按部门/金额/类别筛选）
- 一键通过/驳回，支持填写驳回原因
- 异常报销预警（超限、重复发票）
- 月度/季度报销统计报表

#### 管理员端
- 报销规则配置（各类别上限、可报销项目）
- 用户权限管理
- 审批流配置（几级审批、金额阈值）
- 操作日志查看

#### Agent 智能能力
- 自然语言问答（"差旅报销上限是多少？"）
- 异常检测（重复发票、金额异常）
- 报销政策自动解读

## 五、技术选型

| 层次 | 技术 | 选择原因 |
|------|------|----------|
| 前端 | Streamlit（MVP）/ React（生产） | 快速原型 → 可扩展 |
| 后端 | FastAPI | 高性能，异步支持，自动生成 API 文档 |
| AI 识别 | GPT-4o Vision | 发票 OCR + 结构化信息提取一步到位 |
| 任务队列 | Celery + Redis | 异步处理识别任务，不阻塞用户操作 |
| 数据库 | PostgreSQL | 关系型数据，支持复杂查询和报表 |
| 文件存储 | 阿里云 OSS / MinIO（私有化） | 发票图片安全存储 |
| 缓存 | Redis | API 限流、Session、热点数据缓存 |
| 消息通知 | 企业微信 / 钉钉 Webhook | 审批结果实时通知 |
| 部署 | Docker + Docker Compose | 环境一致，快速部署 |

## 六、核心流程设计

### 6.1 报销提交流程

```
员工上传发票
    ↓
文件存入 OSS，生成唯一 file_id
    ↓
异步任务入队（Celery）
    ↓
GPT-4o 识别发票信息（结构化 JSON）
    ↓
规则引擎校验（金额/类别/合规性）
    ↓
    ├── 校验通过 → 进入审批流
    └── 校验不通过 → 返回问题列表，员工修改
```

### 6.2 审批流程

```
待审批列表
    ↓
直属主管审核（申请后2天内）
    ↓
    ├── 驳回 → 通知员工，说明原因
    └── 通过 → 进入财务审核
              ↓
         财务复核（初步审核后2天内）
              ↓
              ├── 驳回 → 通知员工
              └── 通过 → 安排打款（3天内完成）
                        ↓
                   记录账目，归档发票
```

### 6.3 异常检测（AI 监督机制）

对应痛点：**缺乏有效监督和审查机制**

- **重复发票检测**：对发票号码做唯一性校验，防止重复报销
- **金额异常检测**：同一员工单月同类别报销金额超过阈值自动预警
- **发票真实性**：检测图片是否经过 PS 修改（hash + 元数据校验）
- **定期审查报告**：每季度自动生成异常报销分析报告

```sql

                status, submitted_at, updated_at)

approval_records (id, claim_id, approver_id, action,
                  comment, created_at)

-- 报销规则表
expense_rules (id, category, max_amount, description,
               updated_at, updated_by)
```

## 八、安全设计

| 风险 | 措施 |
|------|------|
| API Key 泄露 | 存密钥管理服务（如阿里云 KMS），不写代码 |
| 发票数据泄露 | OSS 私有 Bucket，前端用临时签名 URL 访问 |
| 未授权访问 | JWT 鉴权 + RBAC 权限模型 |
| 发票被篡改 | 上传时计算 MD5 hash，审批时二次验证 |
| 操作不可追溯 | 所有操作写审计日志，不可删除 |
| 传输安全 | 全链路 HTTPS |

## 九、报销规则配置（可扩展）

对应痛点：**报销政策不清晰**

当前内置规则（可通过管理员后台修改）：

| 费用类别 | 单次上限 | 可报销项目 | 凭证要求 |
|----------|----------|------------|----------|
| 餐饮 | 500元 | 商务餐饮 | 发票+消费明细 |
| 交通 | 2000元 | 出差交通 | 车票/机票/打车发票 |
| 住宿 | 800元/晚 | 出差住宿 | 酒店发票 |
| 办公用品 | 1000元 | 日常办公 | 发票 |
| 其他 | 500元 | - | 发票+说明 |
